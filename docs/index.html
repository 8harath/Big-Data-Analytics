Motivation

The Engineering Problem

Building a pipeline that works once is trivial. Building one that is correct on every subsequent run is not.

Silent Schema Failures
APIs change column names without warning. Pipelines silently break with no error logged, corrupting downstream analytics tables.
Duplicate Records
Without explicit deduplication, re-runs inflate record counts. Two independent layers are required for a correctness guarantee across all failure modes.
Wasteful Full Fetches
Re-fetching the entire dataset on every run is expensive and unscalable. Incremental ingestion with persistent state tracking is required.
Encoding Inconsistency
French accented characters produce mismatched records. Unicode normalization via unidecode must be applied explicitly to every text column before storage.
Reproducibility Gap
"Works on my machine" is not a deployment strategy. All dependencies must be pinned and containerized so any reviewer can reproduce a full pipeline run from scratch.

Goal: A complete, correct, and reproducible streaming ETL pipeline that addresses each of these challenges explicitly and verifiably.

Big Data Tool · Message Queue · Transport Layer

Apache Kafka

KRaft Mode · No ZooKeeper
Why We Selected Kafka
→ Producer and Spark consumer must be completely decoupled — if Spark crashes mid-run, messages must persist without re-calling the API.
→ Apache Spark Structured Streaming has native Kafka integration — no connector code needed, subscribe-and-read in 3 lines.
→ KRaft mode eliminates ZooKeeper, reducing the container count by one and removing the most complex operational dependency in older Kafka setups.
How It Solves The Problem
01 Spark reads startingOffsets: earliest — every restart replays all messages, no data loss possible even across full container restarts.
02 DLQ topic rappel_conso_dlq captures every Pydantic-rejected record with a structured error envelope — invalid data is never silently dropped.
03 Airflow's kafka_data_stream task can finish and succeed independently of Spark — Airflow retries apply to each component separately.

Primary Big Data Engine · Processing Layer

Apache Spark

Core Engine

STRUCTURED STREAMING · VERSION 3.5.0

Why We Selected Spark
→ Only distributed engine with native Kafka source + JDBC sink in a single unified Structured Streaming API — no intermediate steps needed.
→ trigger(once=True) makes streaming behave like a batch job — processes all queued messages, then exits cleanly. Perfect fit for Airflow DAG tasks.
→ Distributed left anti-join deduplication is impossible in plain Python at scale — Spark partitions the comparison across workers, handling any data volume.
How It Solves The Problem
01 25-field StringType schema via from_json() — all-text avoids any type-coercion failure on messy real-world API data. No silent nulls.
02 Left anti-join against existing rappel_conso_table rows on reference_fiche — only absent records pass through. Idempotent across unlimited reruns.
03 foreachBatch append mode via JDBC — writes only new records. DB credentials injected as DockerOperator env vars at runtime. Zero manual config on redeployment.

Big Data Tool · Orchestration · Workflow Scheduler

Apache Airflow

Version 2.7.3

DAG: kafka_spark_dag · @daily · LocalExecutor

Why We Selected Airflow
→ Dependency-ordered task execution with automatic retries — no custom cron scripts needed to enforce: table creation must complete before Kafka runs before Spark runs.
→ DockerOperator support is critical — Spark runs in its own isolated container, and Airflow spawns it via docker-proxy (socat) without being inside the Spark image.
→ Web UI provides a visual DAG graph, task logs, and run history — essential for academic demonstration and debugging failed pipeline runs.
How It Solves The Problem
01 If kafka_data_stream fails, pyspark_consumer never runs — Spark cannot consume stale or partial data. Failure isolation is automatic, not hand-coded.
02 create_target_table runs CREATE TABLE IF NOT EXISTS every DAG run — idempotent database setup, no manual intervention ever required.
03 1 retry with 5s delay per task covers transient API timeouts and Kafka startup delays — the pipeline self-heals from the most common real-world failure modes.

Ingestion Layer

The Kafka Producer

src/kafka_client/kafka_stream_data.py  —  5 sequential steps executed per Airflow run

01
Read State
Load last_processed.json. Returns last successful ingestion date, or defaults to "2000-01-01" on first run, missing file, or corrupt JSON.
02
Paginate API
Fetches 100 records per request. Rollover logic: when offset approaches the 10,000 record date-window cap, the date window advances and offset resets to zero to continue without interruption.
03
Deduplicate — Layer 1
Within-batch dedup via Python dict: { v["reference_fiche"]: v for v in data }.values() — O(n). Handles API returning the same record twice in one date window.
04
Transform
4 transformations: accent normalization (unidecode on 12 columns), column merging (3 pairs), date range regex parsing, and schema reduction to exactly 25 fields.
05
Validate + Publish
Pydantic validates each record. Valid → rappel_conso. Invalid → rappel_conso_dlq with full error envelope. State file updated to max(date) − 1 day.

Data Processing

Transformations & Data Quality

TRANSFORMATION PIPELINE — src/kafka_client/transformations.py

01 — Accent Normalization
IN: "Réseau Générale"
OUT: "Reseau Generale"
unidecode() applied to 12 text columns. Prevents query mismatches from encoding variation in PostgreSQL storage.
02 — Column Merging
risque_col_a + risque_col_b
risques_pour_le_consommateur
3 column pairs merged with newline separator. Preserves both values. Simplifies the schema for downstream queries.
03 — Date Range Parsing
IN: "Vendu depuis le 01/03/2023"
date_debut: "01/03/2023"
Regex (\d{2}/\d{2}/\d{4}) extracts dates from French prose. "depuis le" / "jusqu'" keywords determine start vs end.
04 — Schema Reduction
Raw API: 40+ fields (redundant)
Output: exactly 25 DB_FIELDS
normalize_columns() selects COLUMNS_TO_KEEP + COLUMNS_TO_NORMALIZE. transform_row() adds NEW_COLUMNS. No extras stored.

PYDANTIC VALIDATION + DLQ ROUTING

reference_fiche — non-empty string
Primary key. An empty string passes Python's type check but silently corrupts the PostgreSQL uniqueness constraint.
date_de_publication — YYYY-MM-DD
State file update calls strptime() on this field. Wrong format causes a ValueError and pipeline crash at the state-update step.
Commercialisation dates — DD/MM/YYYY
Output of the regex transformer. A format mismatch means the transform layer produced garbage — caught before the record enters Kafka.
All other fields — Optional[str]
extra="ignore" drops unknown future API fields silently. Valid → rappel_conso. Invalid → rappel_conso_dlq.

"valid: 847 → rappel_conso  |  invalid: 2 → rappel_conso_dlq"

Data Integrity & State Management

Deduplication + Incremental Ingestion

Dedup key: reference_fiche — guaranteed unique in PostgreSQL across all pipeline runs

Layer 1 — Producer (within-batch)
—Runs in Python before any Kafka publish
—{ ref: v for v in data }.values()
—Handles API returning same record twice in one date window
—Scope: current batch only — no knowledge of prior runs
Layer 2 — Spark (cross-run)
—Reads full rappel_conso_table as a static DataFrame
—Left anti-join on reference_fiche: only absent rows pass through
—Distributed across Spark partitions — scales with data volume
—Covers the one-day safety buffer overlap window
Incremental State — last_processed.json
State File Structure
Stores {"last_processed": "YYYY-MM-DD"}. Volume-mounted from host into Airflow container. Survives container restarts and full re-deploys.
First Run
File is empty → falls back to DEFAULT_LAST_PROCESSED = "2000-01-01". Fetches full history. Layer 2 dedup prevents duplicate inserts.
Safety Buffer (−1 Day)
State updated to max(date) − 1 day. Next run re-queries the boundary to catch API indexing delays. Spark anti-join prevents duplicate inserts.
Edge Cases Handled
Corrupt JSON → fallback to default date. API timeout → timeout=30 raises, Airflow retries once. HTTP error → raise_for_status().

Combined guarantee: Regardless of reruns, retries, or the one-day overlap window — every record appears exactly once in rappel_conso_table.

Summary

Conclusion

A complete, correct, and reproducible streaming ETL pipeline — ingesting real product-recall data from the RappelConso API, routing through Kafka, processing with Apache Spark Structured Streaming, storing clean deduplicated records in PostgreSQL, orchestrated daily by Airflow, running entirely inside Docker Compose.

01 Apache Kafka decouples ingestion from processing — KRaft mode, two topics, and replay-from-earliest guarantees zero data loss on any Spark failure.
02 Apache Spark Structured Streaming reads Kafka, parses 25-field schema, performs distributed left anti-join dedup, and writes to PostgreSQL — all in one pipeline.
03 Apache Airflow enforces strict task ordering — table creation → Kafka ingestion → Spark — with automatic retries and DockerOperator isolation.
04 Two-layer deduplication (Python dict + Spark anti-join) provides exactly-once guarantees for stored records regardless of reruns, retries, or API boundary overlaps.
05 Pydantic + DLQ catches schema failures before Kafka. Incremental state with 1-day safety buffer makes every daily run efficient and fault-tolerant.
