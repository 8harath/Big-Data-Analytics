Motivation

The Engineering Problem

Building a pipeline that works once is easy. Building one that is correct on every subsequent run under real-world conditions is hard.

  • Silent schema failures — APIs change column names without warning; pipelines break with no error logged
  • Duplicate records — without explicit deduplication, re-runs corrupt downstream analytics with inflated counts
  • Wasteful full-history fetches — re-fetching everything on every run is expensive and does not scale
  • Text encoding inconsistency — French accented characters produce mismatched records when stored inconsistently
  • Reproducibility gap — "works on my machine" is not a deployment strategy; all dependencies must be pinned and containerized

Goal: Demonstrate a complete, correct, and reproducible streaming ETL pipeline that addresses each of these challenges explicitly.

Transport Layer

Apache Kafka — Message Queue

DESIGN RATIONALE

  • Decoupling — Producer and Spark consumer are independent. Consumer can lag without affecting ingestion.
  • Fault tolerance — If Spark fails mid-run, messages remain in Kafka. No re-fetch from the API is required.
  • KRaft mode — ZooKeeper eliminated. Combined controller + broker in a single node. Modern, simpler deployment.
  • Dual topics — rappel_conso (main pipeline) and rappel_conso_dlq (invalid records).

LISTENER CONFIGURATION

PLAINTEXT://kafka:9092
Internal — container-to-container communication within Docker network
CONTROLLER://kafka:9093
KRaft consensus communication — never advertised to clients
EXTERNAL://localhost:9094
Host-machine access — mapped via Docker port publish

Producer auto-fallback: tries kafka:9092 first, falls back to localhost:9094 if unreachable

Left anti-join against existing rappel_conso_table. Only records absent from the database are written. Distributed, runs before any JDBC call.

Data Processing

Transformation Pipeline

src/kafka_client/transformations.py  ·  Applied to every record before Kafka publish

01 — ACCENT NORMALIZATION
Input:  "Réseau de distribution Générale"
Output: "Reseau de distribution Generale"
unidecode() applied to 12 text columns. Ensures consistent ASCII storage. Prevents mismatched queries due to encoding variation.
02 — COLUMN MERGING
risque_col_a + risque_col_b (2 fields)
risques_pour_le_consommateur (1 field)
3 pairs merged: risks, health recommendations, additional info. Newline separator preserves both values. Simplifies downstream queries.
03 — DATE RANGE PARSING
Input:  "Vendu depuis le 01/03/2023..."
date_debut: "01/03/2023" / date_fin: ""
Regex (\d{2}/\d{2}/\d{4}) extracts dates. French keywords "depuis le" / "jusqu" determine start vs end. Produces structured date fields from prose.
04 — SCHEMA REDUCTION
Raw API: 40+ fields (some redundant)
Output: 25 DB_FIELDS (8 kept + 12 normalized + 5 new)
normalize_columns() selects exactly COLUMNS_TO_KEEP + COLUMNS_TO_NORMALIZE. transform_row() adds NEW_COLUMNS. Result is a clean, consistent 25-field dictionary.

Data Quality Layer

Pydantic Validation + Dead-Letter Queue

WHAT WE VALIDATE

reference_fiche — non-empty string
Primary key. Empty string passes Python's type check but silently corrupts the PostgreSQL uniqueness constraint.
date_de_publication — YYYY-MM-DD
State file update calls strptime() on this field. Wrong format → ValueError → pipeline crash without state file update.
Commercialisation dates — DD/MM/YYYY
Output of the regex transformer. A mismatch indicates the transformation layer produced garbage — caught here before it reaches Kafka.
All other fields — Optional[str]
API does not guarantee presence. extra="ignore" silently drops future API fields, preventing breakage on new columns.

ROUTING LOGIC

VALID RECORD

→ Serialized to JSON → published to rappel_conso

INVALID RECORD

→ Wrapped in DLQ envelope → published to rappel_conso_dlq

Envelope: {original_record, validation_errors, failed_at, source_topic}

End of run: "valid: 847 → rappel_conso | invalid: 2 → rappel_conso_dlq"

Data Integrity

Two-Layer Deduplication Strategy

Key: reference_fiche — the unique recall notice identifier

LAYER 1 — PRODUCER (within-batch)
Runs in Python before any Kafka publish
{ v["reference_fiche"]: v for v in data }.values() — O(n) set deduplication
Handles API returning the same record twice within one date window
Reduces Kafka topic size — no duplicate messages stored
Scope: current batch only; does not know about prior runs
LAYER 2 — SPARK (cross-run)
Runs inside the Spark container, after Kafka read
Reads full rappel_conso_table from PostgreSQL into a static DataFrame
Left anti-join on reference_fiche: only rows absent from DB pass through
Distributed across Spark partitions — scales with data volume
Covers daily overlap window from the one-day safety buffer

Combined guarantee: Regardless of how many times a record enters the pipeline — through reruns, retries, or the overlap window — it appears exactly once in rappel_conso_table.

State Management

Incremental Ingestion

HOW IT WORKS

STATE FILE — last_processed.json
Stores {"last_processed": "YYYY-MM-DD"}. Volume-mounted from host into Airflow container. Survives container restarts.
FIRST RUN
File is {} (empty). Falls back to DEFAULT_LAST_PROCESSED = "2000-01-01". Full historical dataset fetched. May take several minutes.
SAFETY BUFFER (-1 DAY)
State is updated to max(date) − 1 day. Next run re-queries boundary records to catch API indexing delays. Spark anti-join prevents duplicate inserts.
ROLLOVER LOGIC
API caps at 10,000 records per date window. When offset approaches cap, date window advances to last record's date − 1 day and offset resets.

EDGE CASES HANDLED

  • Corrupt JSON file — json.JSONDecodeError caught, logged, falls back to default date
  • Missing file — Path.exists() check, returns default date
  • API timeout — requests.get(timeout=30) raises, Airflow retries task once
  • API HTTP error — raise_for_status() propagates to Airflow retry

Reset procedure: Set file to {} for full re-fetch. Anti-join prevents all duplicates.

Lessons Learned

Challenges & Solutions

The repository had six reproducibility problems that prevented end-to-end execution. Each was diagnosed and fixed.

ProblemRoot Cause & Fix
DAG not loading in AirflowCompose file mounted ./dags (non-existent) instead of ./airflow_resources/dags. Fixed volume mount path.
DockerOperator image not foundDAG referenced rappel-conso/spark:latest but no build step existed. Added spark-app build service to docker-compose.yml.
Database connection failuresPython scripts and Compose used different DB credentials. Centralized all settings as environment variables read via constants.py.
State file path errors in containersRelative path broke inside Docker. Made path configurable via LAST_PROCESSED_PATH env var.
Pipeline fails on first runTarget table had to be created manually. Moved CREATE TABLE to the first Airflow task with IF NOT EXISTS.
DockerOperator ImportErrorapache-airflow-providers-docker not installed in Airflow image. Added to requirements-airflow.txt.

Reflection

Limitations & Future Work

CURRENT LIMITATIONS
Single-node Spark in local[*] mode — no cluster-level parallelism
Airflow metadata and pipeline output share one PostgreSQL instance
No automated test suite for transformation functions
Spark Maven packages downloaded on every run — requires internet and adds startup latency
No CI/CD pipeline for automated build and validation
No schema evolution handling if API adds or renames columns
PLANNED IMPROVEMENTS
Pre-bake Spark JARs into the Docker image — eliminates runtime Maven download
dbt for post-load SQL transformations with built-in test assertions
ClickHouse or DuckDB as an analytical store — 10–100× faster aggregation queries
pytest suite for all transformation functions with edge-case coverage
Kubernetes deployment manifests for horizontal scaling
Apache Superset or Metabase dashboard for recall trend visualisation

Summary

Conclusion

We built a complete, correct, and fully reproducible streaming data pipeline that ingests real product-recall records from a French government API, routes them through Kafka, processes them with Apache Spark Structured Streaming, and stores clean, deduplicated records in PostgreSQL — orchestrated daily by Airflow and running entirely in Docker Compose.

01Apache Spark is the primary Big Data tool — Structured Streaming, schema parsing, distributed anti-join deduplication, JDBC write.
02Two-layer deduplication (producer set + Spark anti-join) provides exactly-once guarantees for stored records regardless of reruns.
03Pydantic validation catches schema failures before they enter Kafka, routing invalid records to a dead-letter topic with structured error metadata.
04Incremental state management via a JSON file with a one-day safety buffer ensures daily runs are efficient and fault-tolerant.
05Docker Compose with pinned image versions makes the full stack reproducible on any machine with no manual dependency setup.
